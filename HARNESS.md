# HARNESS.md

这个文件是 `my-agent` 仓库的项目操作手册。

如果你是进入这个仓库工作的 agent，请把它当成高优先级项目约定来遵守。

## 项目目标

- 这个仓库要做的是一个本地编码 agent CLI。
- 用户希望最终体验接近 `claude` / `codex` 这类工具：进入终端后直接运行 `jarvis`。
- 代码要优先服务“真实可用的产品感”，而不是只做 demo。

## 交付习惯

- 每次完成一个明确的用户请求后：
  - 跑相关验证
  - 更新 `CHANGELOG.md`
  - 创建 1 个 Git commit
- 如果这轮只是讨论、调查、没有改文件，不要强行 commit。
- 如果工作区里有你没改过的用户变更，不要把不相关内容混进 commit。
- 一个用户请求尽量对应一个 commit，commit message 要简短、明确。

## 总结格式

- 每次更新 `CHANGELOG.md` 时，至少写清楚：
  - 改了什么
  - 为什么改
  - 怎么验证
- 最终回复用户时，也应说明：
  - 这轮完成了什么
  - 改了哪些关键文件
  - 是否已经 commit

## 代码与产品方向

- 保持 CLI 体验简单、直接、像工具而不是脚本。
- 优先增强这些能力：
  - 会话体验
  - Git 可观察性
  - 安全边界
  - 编辑能力
  - 可安装/可启动体验
- 文档要和代码行为保持一致。
- 如果任务涉及长期演进方向、架构路线或性能优化，先查看仓库根目录的 `way-to-claw-code.md`。

## Git 约定

- 默认分支信息、diff、变更状态对用户是重要信息。
- 能自动总结时，优先把一轮工作的成果体现在：
  - `CHANGELOG.md`
  - Git commit message
